## 1. Introduction

Large Language Models (LLMs) are trained on vast datasets to perform a wide array of diverse tasks, among which machine translation has become prominent. Leveraging their massive multilingual capabilities and extensive world knowledge, these systems operate either as general-purpose models capable of translation, or as specialized models fine-tuned specifically for translation and its associated subtasks.

The changing balance between translation paradigms is clearly reflected in recent evaluation campaigns. Data from the annual Conference on Machine Translation (WMT) general translation shared tasks demonstrates a rapid paradigm shift: LLMs are increasingly being adopted for translation tasks and are actively replacing traditional Neural Machine Translation (NMT) as the dominant methodology (see Table 1). This trend reflects a broader shift in research priorities and system architecture.

However, the transition from NMT to LLMs is not merely a mechanical replacement of one architecture by another. Instead, it represents a fundamental shift in the underlying logic of machine translation development. The field is moving away from highly specialized, task-specific translation engines toward flexible, instruction-driven, multilingual systems. Consequently, translation performance no longer depends solely on the core architecture, but is highly influenced by:

* Model scale and parameter size
* The diversity of the training data mixture
* Advanced prompting strategies
* The specific evaluation setup

This growing complexity makes direct system comparison significantly more challenging. Differences in translation quality are increasingly shaped by benchmark selection, domain specificities, and task formulation, rather than architectural differences alone.

## 2. Instruct LLMs and Translation-Specific Models

A primary and important distinction among LLMs used for translation is between instruct models and those tailored specifically for translation.

* General-purpose **instruct LLMs** are capable of performing a wide variety of tasks depending on the instruction or prompt provided: answering questions, engaging in dialogue, summarizing text, etc. Among these tasks is text translation, for which a specific prompt must be supplied. Examples of general models include ChatGPT, Gemini, Llama, and others.
* **Translation-specific LLMs** are models designed exclusively for translation and cannot perform other tasks (technically, you can prompt them to do so, but they will not provide a satisfactory response). Many of these models are derived from general LLMs that have undergone *fine-tuning* for translation by being exposed to parallel corpora. Generally, these specific models rely on predefined prompts for translating, which stem from how the parallel corpus was structured during the fine-tuning phase. In addition to translation, these specialized models can handle other translation-related tasks, such as error correction or automatic post-editing. Examples of translation-specific models include TowerInstruct and SalamandraTA.

Another crucial factor to consider when using LLMs for translation is whether the LLM "knows" the language in question—that is, whether the training corpus contained a sufficient volume of texts for that specific language pair. If it did not, the system might still be able to perform translations, but they will be *zero-shot* translations. A zero-shot translation occurs when a model translates between two languages without having been explicitly trained on that specific language pair during its supervised training phase.

For this reason, it is vital to consult the detailed model cards of the models you intend to use to find further specifics regarding available languages and prompt recommendations.

### 3.1. Specific Prompts for Translation-Specific LLMs

**3.1.a. TowerInstruct**

The model card for [TowerInstruct](https://huggingface.co/Unbabel/TowerInstruct-7B-v0.1) specifies that the prompt must follow this structure:

```
Translate the following text from [Source_language] into [Target_language].
[Source_language]: [The source language text to translate].
[Target_language]:
```

For instance, if we input:

```
Translate the following text from English into Spanish.
English: This is a sample sentence to be translated.
Spanish:
```

The TowerInstruct model will return:

```Este es un ejemplo de oración que se debe traducir.```

It should be noted that the system will also respond successfully to similar prompts, such as:

``
English: This is a sample sentence to be translated.
Spanish:
``

You can test any of these prompts and observe the outcome. Depending on the model and the prompt, the system might return additional explanations alongside the translation. For instance, if you test the following prompt in ChatGPT:

`Given the following source text (in English): "This is a sample sentence to be translated", a good Spanish translation is:`


It will return (test conducted on 07/18/2025):

```

"This is a sample sentence to be translated"

is:

"Esta es una frase de muestra para traducir."

Alternative natural variant:
"Esta es una oración de ejemplo para traducir."

Explanation:
"Esta es" = "This is" (feminine form because frase or oración is feminine)

"una frase de muestra" / "una oración de ejemplo" = "a sample sentence"

"para traducir" = "to be translated" (expressing purpose using the infinitive)

Both versions are grammatically correct. The choice between frase and oración may depend on the regional or stylistic preference:

Frase is more common in informal usage and refers to a phrase or sentence.

```

While this may be interesting for a human reader, if we want to make automated queries to the LLM via an API—for example, within a computer-assisted translation (CAT) tool—we will require the LLM to return the translation and nothing else. This can be achieved by slightly modifying the prompt. For example, the prompt:

```
Provide only the translation and nothing else. Given the following source text (in English): "This is a sample sentence to be translated", a good Spanish translation is:
```

Returns exclusively:

```
Esta es una frase de muestra para traducir.
```

## 5. Ollama

### 5.1. Introduction

Ollama is a platform that enables large language models (LLMs) to run locally on any computer. It provides a straightforward interface to download, execute, and interact with models such as LLaMA, Mistral, and Gemma, among others, without relying on cloud-based services. Ollama streamlines the use of these models for tasks like text generation, summarization, question answering, and more, either via a command-line interface or through its local API.

It is designed to be lightweight, user-friendly, and cross-platform, supporting Windows, Linux, and macOS.

Comprehensive information about Ollama can be found at [https://ollama.com/](https://ollama.com/).

### 5.2. Installing Ollama

**Windows:**

Download the Windows installer from [https://ollama.com/download/windows](https://ollama.com/download/windows) and run the executable file.

**Linux:**

From a Linux terminal, type:

`sudo curl -fsSL https://ollama.com/install.sh | sh`

**Mac:**

It can be installed in two ways via the terminal. You can run:

`curl -fsSL https://ollama.com/install.sh | sh`

Or alternatively:

`brew install ollama`

It is also possible to download the installer (.dmg) from [https://ollama.com/download/mac](https://ollama.com/download/mac).

### 5.3. Installing a Model

Once Ollama is installed, you can download as many models as you require. Several factors should be kept in mind regarding this:

* The Ollama homepage ([https://ollama.com/](https://ollama.com/)) features a model search engine. Simply enter the name to see matches.
* Many models are distributed in various parameter sizes. You must consider your available disk space and your computer's processing power to download sizes appropriate for your hardware.
* You can install models directly through Ollama or via the application proposed in the next section.

To install a model using Ollama, such as the deepseek-r1:8b model, you can run the following command in a Terminal or Command Prompt window:

`ollama run deepseek-r1:8b`

In the role:user field, you will enter your prompt (the request sent to the system). Clicking Send transmits it to the model, and the answer will be displayed in the Response box.

As you can see, there are three fields where a request can be structured: role:system, role:user, and role:assistant. Although you can operate solely using role:user, the other roles are highly beneficial for crafting more precise prompts in certain models. The system, user, and assistant roles define who is speaking and for what purpose:

**role: system**
* Identity: The "invisible" voice that configures the assistant's behavior.
* Purpose: Defines the initial context, operating constraints, or stylistic guidelines for the assistant.

Example:

`{ "role": "system", "content": "You are a linguistics expert who responds in Catalan." }`

* Importance: It is typically established only once (at the beginning) and does not change during the conversation unless intentionally restarted or modified.

**role: user**

* Identity: The person interacting with the model (the end-user).
* Purpose: Represents the actual messages, queries, or commands addressed to the assistant.

Example:

`{ "role": "user", "content": "Què és una gramàtica generativa?" }`

**role: assistant**

* Identity: The model responding to the query (e.g., ChatGPT).
* Purpose: Contains the responses generated by the system based on the preceding conversation history.

Example:

`{ "role": "assistant", "content": "La gramàtica generativa és una teoria lingüís..."}`

## Appendix 1. Techniques for Reducing LLM Model Sizes 

Important note: This appendix was authored by ChatGPT on 07/17/2025 in response to the following prompt: Can you explain to me how the size of an LLM is reduced?

Reducing the size of a Large Language Model (LLM) involves applying various techniques that decrease the number of parameters or the overall computational demand without excessively sacrificing model quality or accuracy. The primary strategies are detailed below:

### 1. Pruning

**Definition**: Eliminates less relevant weights or neurons after training has concluded.

**Types**:

* Unstructured pruning: Individual connections are removed in a sparse pattern.
* Structured pruning: Entire neurons, layers, or channels are removed systematically.

**Advantage**: Decreases model size and enhances inference speed.

### 2. Knowledge Distillation

**Definition**: Training a smaller model (the "student") to mimic the behavior of a larger, more complex model (the "teacher").

**Mechanism**: The student model is trained on the outputs (probabilities, logits) generated by the teacher. Consequently, the smaller model learns not just the final response but also the intermediate probability distributions.

**Outcome**: A compact model that retains highly competitive performance.

### 3. Quantization

**Definition**: Reducing the numerical precision of the model's weights—for instance, converting them from 32-bit floating-point (float32) to 8-bit integers (int8).

**Types**:

* Post-training quantization: Applied after the model has already been trained.
* Quantization-aware training: The model is trained from the ground up with the awareness that it will eventually undergo quantization.

**Advantages**:

* Conserves storage space.
* Maximizes efficiency on specialized hardware (TPUs, GPUs).

### 4. Weight Sharing

**Definition**: Reusing a small set of weights across different parts of the model architecture.

**Example**: Utilizing identical weights across all attention or feed-forward layers.

**Outcome**: Significantly lowers the total number of unique parameters.

### 5. Low-Rank Factorization

**Definition**: Approximating large matrices within the model (such as transformer weights) by decomposing them into products of lower-rank matrices.

**Example**: Matrix W ≈ A × B, where A and B represent significantly smaller matrices.

**Advantage**: Fewer parameters and reduced computational operations during inference.

## 6. Tensor Decomposition and Sparsity

**Tensor Decomposition**: Conceptually similar to matrix factorization but applied to tensors (multidimensional structures).

**Sparsity**: Exploiting the fact that many weights can be set to zero or near-zero values.

This reduces both memory overhead and runtime calculations.

### 7. Adapter Layers / LoRA (Low-Rank Adaptation)

Instead of modifying the entire model architecture, lightweight layers (adapters) are appended, or low-rank updates (LoRA) are inserted, which are trained for specific tasks. In doing so:

* The original model remains frozen and untouched.
* Only the lightweight adaptive components are stored and executed.
* This is exceptionally useful for minimizing the footprint of customized or multilingual models.

### Combination of Techniques

Most practical implementations combine multiple techniques simultaneously:

e.g., distillation + quantization + pruning.

## Appendix 2. Using Ollama via the Terminal

This appendix outlines how to interact with Ollama through the terminal interface to manage models and run prompts directly.

### 1. Listing Installed Models
To view which models have already been downloaded and installed locally, run:

`ollama list`

This command will output a list displaying the name, size, installation date, and tags of all available models.

### 2. Installing New Models

To download a model from the official Ollama repository, use:

`ollama run <model>`

For example, to install and interact with the llama3 model:

`ollama run llama3`

The model will download upon its first invocation. Subsequent executions will load it directly from your local hardware.

Alternatively, you can pull the model without launching it immediately by running:

`ollama pull llama3`

### 3. Executing Prompts via the Terminal

Once the model is successfully installed, you can initiate a conversation directly:

`ollama run llama3`

Thereafter, type your prompt. For example:

`> What is Renaissance humanism?`

The model will generate a response within the terminal. You can continue sending messages as long as the interactive session remains active.

### 4. Exiting Interactive Mode

To terminate Ollama's interactive conversation mode, press:

`Ctrl + C`

This will close the session and return you to your standard command-line prompt.

### 5. Executing Single-Line Prompts (Scripting Mode)

If you wish to automate a question/answer sequence in a single command, you can pipe it:

`echo "Summarize the plot of 'Don Quixote'" | ollama run llama3`

Or you can use curl to query Ollama's local API (port 11434):

curl http://localhost:11434/api/generate -d '{
"model": "llama3",
"prompt": "Summarize the plot of Don Quixote"
}'


## Appendix 3. Advanced Parameters in Ollama

**temperature**

* Mechanism: Controls the degree of randomness in the generated responses.
* Low value (e.g., 0.1): Leads to more predictable, consistent, and repetitive outputs.
* High value (e.g., 0.9): Fosters more creative, diverse, and varied outputs.
* Recommendation: A range of 0.7–1.0 is standard for text generation requiring stylistic variety.

**top_p** * Mechanism: Implements nucleus sampling, limiting word selection to a pool of top candidate words whose cumulative probability matches the specified percentage.
* Value of 1.0: The model considers all possible words.
* Value of 0.9: Limits options to the words that collectively account for the top 90% of probability.
* Recommendation: Helps balance text coherence with creative flexibility.

**top_k**

* Mechanism: Hard-limits choice selection to the 'k' most probable words at each generation step.
* Low value: Yields highly controlled, less creative text.
* High value: Permits wider vocabulary choices, resulting in increased creativity.
* Recommendation: Typically utilized in tandem with top_p, or as an alternative to it.

**repeat_penalty**

* Mechanism: Penalizes the recurrence of identical words or phrases within the generated output.
* Value > 1: Actively discourages the model from repeating the same terms continuously.
* Value < 1: Allows for a higher degree of repetition.
* Recommendation: A value of 1.1 is appropriate if repetitive patterns become problematic.

**seed**

* Mechanism: Configures a specific random seed to ensure deterministic, reproducible results.
* No value (blank): Produces distinct outcomes with every execution.
* Numeric value specified: Generates the exact same output across identical runs.

**num_predict**

* Mechanism: Defines the maximum token length (words or word fragments) the model is allowed to generate.
* Value of -1: Removes constraints, allowing generation until the model naturally concludes.
* Positive integer value: Enforces a strict upper length boundary.
* Recommendation: Useful for constraining responses if a model tends to be overly verbose.

**as_json**

* Mechanism: Dictates whether the final output should be structured as a JSON object.
* False: Standard text string output.
* True: Returns structured JSON data (e.g., { "response": "...text..." }).
