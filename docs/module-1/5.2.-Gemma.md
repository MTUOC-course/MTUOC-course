## 1. Introducció

Gemma is a family of lightweight, state-of-the-art open models developed by Google DeepMind and other teams across Google. Named after the Latin word for "precious stone," Gemma models are built from the same core research, technology, and infrastructure used to create Google's massive proprietary Gemini models. However, unlike Gemini, Gemma models have "open weights," meaning developers are free to download, customize, and run them locally on their own hardware.

The family has evolved rapidly, expanding from purely text-based models to powerful multimodal assistants. The ecosystem is categorized by specific generations and specialized variants:
Core Generations & Sizes

* Gemma 4 (Latest): The newest generation, engineered specifically for advanced reasoning and agentic workflows. They feature a massive context window (up to 128K or 256K tokens) and support over 140 languages. They are highly unique because they are multimodal and encoder-free, allowing text, images, and native audio inputs to flow directly into the LLM backbone. They come in several scales built for varying hardware, including E2B, E4B, 12B (optimized for 16GB RAM laptops), and larger 26B / 31B variants.
* Gemma 2: The second generation, which vastly improved inference efficiency and intelligence-per-parameter. Available in 2B, 9B, and 27B sizes.
* Gemma (1st Gen): The original lightweight models released in 2B and 7B sizes, trained primarily on English web documents, code, and math.

*Specialized Variants:**

Google also provides bespoke versions of Gemma optimized for specific tasks:

* CodeGemma: Fine-tuned specifically for rapid code completion and natural language code chat across languages like Python, Java, C++, and Rust.
* TranslateGemma: Tailored specifically to break down communication barriers across 55 languages.
* ShieldGemma 2: Specialized safety content moderation models designed to filter and evaluate user inputs and AI outputs against harm categories.
* DiffusionGemma: An experimental Mixture-of-Experts (MoE) model that uses text diffusion rather than token-by-token processing, pushing local generation speeds up to 4x faster on modern GPUs.

Because of their efficiency, smaller variants like the one we're going to test (gemma4:e4b) are heavily favored by the developer community for offline, on-device tasks like local translation and private data synthesis.

If you want to learn more about Gemma, you can read:

[Team, G., Mesnard, T., Hardin, C., Dadashi, R., Bhupatiraju, S., Pathak, S., ... & Kenealy, K. (2024). Gemma: Open models based on gemini research and technology. arXiv preprint arXiv:2403.08295.](https://arxiv.org/abs/2403.08295)

## 2. Testing Gemma in a Pyhton script

In this example we will use Gemma in a Pyhton script using Ollama. Make sure that you have Ollama installed in your system and the Pyhton library installed in your environment (`pip install ollama`).

The following script does the job:

```
import ollama

# --- ENVIRONMENT CONFIGURATION ---
sl_segment_input = "Hello, how are you doing today? I hope you are having a great time."

# --- OLLAMA SERVER & MODEL SETTINGS ---
MODEL = "gemma4:e4b"
HOST = "http://localhost:11434"
TIMEOUT = 5  # Note: Connection timeout is usually managed at the client level

# --- LLM GENERATION PARAMETERS (Ollama Options) ---
generation_params = {
    "temperature": 0.0,
    "num_predict": 128,
    "num_ctx": 2048,
    "repeat_penalty": 1.2,
    "top_k": 40,
    "top_p": 0.9,
    "seed": 42,
    "stop": ["\n", "###"]
}

# --- LLM PROMPT AND RESPONSE PARSING ---
# We prepare the prompt by replacing the {SLsegment} variable
prompt_template = f"Translate the following text from English into Spanish. Provide only the translation.\nEnglish: {sl_segment_input}"

def executePrompt(prompt):
    try:
        # We initialize the client with the specified host
        client = ollama.Client(host=HOST)
        
        # We make the call to the generation API (generate)
        # You could also use client.chat if it were a message structure
        response = client.generate(
            model=MODEL,
            prompt=prompt,
            options=generation_params
        )
        
        # We extract the text from the response
        output_text = response['response']
        
        # --- POST-PROCESSING (json_key & regex_pattern) ---
        # In this case, since json_key is "None" and regex_pattern is None,
        # we directly print the raw result of the response.
        print("--- Gemma 4 answer ---")
        print(output_text)
        
    except Exception as e:
        print(f"Error in connection or generation: {e}")

if __name__ == "__main__":
    prompt="Translate the following text from English into Spanish. Provide only the translation.\nEnglish: This an example of translation using a large language model."
    executePrompt(prompt)
```
If you run this script, the following will be printed in the output:

```
--- Gemma 4 answer ---
Este es un ejemplo de traducción utilizando un modelo de lenguaje grande.
```

## 3. Starting Gemma in MTUOC-server

The easiest way is to use MTUOC-server-manager and download the recipe recipe-ollama-Gemma4.yaml.

This will download the following files:

**config-server-ollama-Gemma4.yaml**

```
MTengine: Ollama
#download, Marian, Eole, OpusMT, NLLB, HuggingFace, Ollama, GoogleT, DeepL

#path to memory or None
  Path: None  
  sl: es
  tl: en
  
MTUOCServer:
  port: 8000
  type: MTUOC
  #one of MTUOC, Moses, ModernMT, OpenNMT, NMTWizard
  verbosity_level: 1
  #specific configuration when acting as ONMT server

  timeout: 45

dopreprocess: False
preprocess_config: None
dotokenization: False
tokenization_config: None
dopostprocess: False
postprocess_config: None
```

And the file:

**config-model-ollama-Gemma4.yaml**

```
# --- OLLAMA SERVER & MODEL SETTINGS ---
ollama_settings:
  model: "gemma4:e4b"                  
  host: "http://localhost:11434"    # URL where the Ollama server is running
  timeout: 5                         # Connection timeout in seconds

# --- LLM GENERATION PARAMETERS (Ollama Options) ---
# These parameters control the behavior of the text generation process
generation_params:
  temperature: 0.0      # 0.0 is deterministic (greedy). Higher values (up to 1.0) are more creative
  num_predict: 128      # Maximum number of tokens to generate (-1 for infinite)
  num_ctx: 2048         # Size of the context window (model memory)
  repeat_penalty: 1.2   # Penalty to prevent the model from repeating the same tokens
  top_k: 40             # Limits the next token selection to the top K most likely tokens
  top_p: 0.9            # Nucleus sampling: filters tokens by cumulative probability
  seed: 42              # Set a seed for reproducible results
  stop: ["\n", "###"]   # List of sequences that will stop the generation process

# --- LLM PROMPT AND RESPONSE PARSING ---
prompt_settings:
  # The template uses {P[0]}, {P[1]}, etc., corresponding to the input file columns
  prompt_template: |
    Translate the following text from English into Spanish. Provide only the translation.     English: {SLsegment} 
  
  # If the model returns a JSON string, specify the key to extract. Use "None" otherwise.
  json_key: "None"
  
  # Regular expression to clean the output (e.g., taking only the first line)
  # '^(.+?)(?=\\n|\n|$)' extracts everything until the first newline
  regex_pattern: None
```

Please, note that in this case, as we're using Ollama, we don't have a folder with the model in the MTUOC-server folder. By default they are located in the following folders depending on the operating system:

* Linux: `/usr/share/ollama/.ollama/models`
* Windows: `C:\Users\<Your-Username>\.ollama\models`
* Mac: `~/.ollama/models`
